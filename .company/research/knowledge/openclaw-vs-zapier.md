# OpenClaw vs Zapier: 役割分担の考え方

## 概要

OpenClaw と Zapier（または n8n、Make などのノーコード自動化ツール）は**競合ではなく補完関係**にある。
導入支援では、どちらに何をやらせるかの設計が重要。

## 役割分担の基本原則

| 処理の種類 | 向いているツール | 理由 |
|-----------|----------------|------|
| トリガー → アクションの単純ルーティング | Zapier | LLM不要、低コスト |
| フォーマット変換・通知・データ連携 | Zapier | 判断不要、高速、安価 |
| 内容を読んで判断が必要な処理 | OpenClaw | LLMの強みが活きる |
| コードベース・ドキュメント参照が必要 | OpenClaw | ローカルファイル操作 |
| 複数ステップ（調べる→判断→実行）の処理 | OpenClaw | エージェント的な推論が必要 |

## 実例: バグ報告フロー

```
Slack #bugs に投稿
    ↓
[Zapier] → GitHub Issue を自動作成（判断不要）
    ↓
[OpenClaw] → コードを調べて原因特定・一次回答をコメント（判断必要）
```

Zapierで処理できる部分はZapierに任せることで、OpenClawの稼働コスト（LLMトークン代）を削減できる。

## コスト視点での判断基準

- **OpenClawに任せると**: 1回あたり数十〜数百円のLLMコストが発生
- **Zapierに任せると**: 月額固定費（Zapierプラン費用）の範囲内、1回あたりのコストはほぼゼロ

→ ルーティング・通知・データ変換はZapierで処理し、**OpenClawは「判断が必要な業務」に集中させる**のがコスト最適。

## 導入支援での推奨アプローチ

1. 現行の業務フローを洗い出す
2. 各ステップが「判断不要（ルール通り）」か「判断必要（文脈理解・推論）」かを分類
3. 判断不要 → Zapier/Make等へ移行
4. 判断必要 → OpenClawへ委譲
