# OpenClaw: チャンネルポリシーとグループチャット運用

## groupPolicy: allowlist（推奨）

デフォルトでは全チャンネルのメッセージに反応してしまい、邪魔になる。
`groupPolicy: "allowlist"` で必要なチャンネルだけを許可するのが正解。

### 設定後の運用
- 新チャンネルを追加したいときは allowlist に手動追加
- requireMention: true を組み合わせると「メンションされた時だけ反応」になる
- allowBots: true でワークフロー bot からのメッセージにも反応できる

### 設定例（openclaw.json イメージ）
```json
{
  "slack": {
    "groupPolicy": "allowlist",
    "allowedChannels": ["C07UKTMLE1G", "C0AMMKLQZ2B"],
    "requireMention": true
  }
}
```

## チャンネルごとの役割分担

チャンネルごとにエージェントの振る舞いを変えると運用がスムーズになる。

### パターン例（Bocek社内）

| チャンネル種別 | 振る舞い |
|-------------|--------|
| times（生存確認） | 自由投稿、メンション不要 |
| request | タスク受付、即対応 |
| バグ報告 | コード調査→PR作成まで自動化 |
| CS報告 | 一次対応、仕様確認 |
| 運用相談 | 設定変更はオーナー確認必須 |
| times_${member} | メンバーごとにトーンを変える |

### チャンネル特有のペルソナ設定
AGENTS.md や SOUL.md にチャンネルIDと対応するトーンを記載することで、
チャンネルごとに異なる話し方を実現できる。

## グループチャットでの発言ルール

全メッセージに反応しない。人間と同じ感覚で「貢献できるときだけ発言」が正解。

### 発言すべきタイミング
- 直接メンションされた
- 質問に回答できる
- 重要な誤情報を訂正できる

### 黙るべきタイミング
- 雑談・banter が盛り上がっている
- 誰かがすでに回答した
- 「いいね」程度の反応しかできない

→ 絵文字リアクション（👍 😂 🤔）を活用して、チャットを汚さずに存在感を示す。

## Slackスレッド返信の絶対ルール

**`threadId` なしで `message(action=send)` を呼ぶことは絶対禁止。**

### threadId の決め方
1. `topic_id` がある → それを使う（既存スレッドへの返信）
2. `topic_id` がない → `message_id` を使う（新規スレッド作成）

この1点を守るだけで、チャンネルへの誤投稿が100%防げる。
