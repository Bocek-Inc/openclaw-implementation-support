# OpenClaw コード調査・バグ修正パターン集

開発チームへの導入支援で役立つ、OpenClawのコード調査・バグ修正ユースケース事例集。

## パターン1: CIテスト失敗のクロスリポ調査

### ユースケース
「mainのE2Eテストが落ちている。原因を調べて」という一言で、CI調査からコード特定まで完全自動化。

### 実例（2026-04-22: taskhub-neo-frontend）
- **状況**: GitHub ActionsのE2Eテストが `tasks.new.spec.ts` 系で一括失敗
- **調査フロー**:
  1. GitHub CI実行ログを取得し、失敗開始タイミングを特定
  2. 失敗開始直前にmainへマージされたPR一覧を `gh` CLIで取得
  3. 差分のあったPR #3477 (`globalSettingsSchema` 変更) を特定
  4. TypeScriptのspread演算子（`...globalSettingsSchema.entries`）経由で `tasks` ルートの `editAppSchema` に影響が波及していることをコード追跡

- **ポイント**: PR #3477のE2Eテスト自体はすべてSUCCESS。PRブランチは `agent-app` 関連ファイルのみ変更のため、`tasks.new.spec.ts` が対象外だった。スキーマのspread共有という設計上の落とし穴。

### 所要工数比較
| 作業者 | 所要時間 |
|--------|---------|
| 熟練エンジニア | 1〜3時間 |
| OpenClaw | 数分（Slack投稿→調査結果報告） |

### 導入提案トーク
> 「mainのCI失敗調査は毎回エンジニアが手を止めて行う作業です。OpenClawなら『調査して』の一言で、CIログ・git履歴・コードを横断して根本原因を特定し、チームのSlackスレッドに報告できます」

## パターン2: TypeScript スキーマspread波及パターンの検知

### 問題の本質
TypeScriptで `...schema.entries` のようにスキーマをspreadしていると、片方のスキーマ変更がもう一方に影響する。この影響はPR単体のE2Eでは検知できない（変更ファイルに影響先のルートが含まれないため）。

### OpenClawによる気づき
コードを追跡した結果、「このPRのE2Eは通っているが、mainブランチで落ちる理由はスキーマspread経由の波及」という設計上の問題点まで指摘できる。

### 活用アドバイス
- PR作成時の「波及チェック依頼」をOpenClawに任せる運用が効果的
- 「このスキーマを使っているファイルを全部探して」+ 「影響がありそうなテストを列挙して」

## パターン3: Slack投稿→コード修正→PR作成の完全自動化

### ユースケース
「バグ報告チャンネルにスクショ付きで報告」→「OpenClawがコードを調査して修正ブランチを作成」

### 実例（2026-04-22: Keycloakログイン画面バグ）
- **バグ内容**: SPのChromeパスワードマネージャー使用時、パスワード入力ステップでemailフィールドが別アカウントのアドレスで上書きされる
- **OpenClawの対応**:
  1. Keycloakカスタムテーマのコードを探索
  2. `LoginForm.tsx` の `EmailField` に `readOnly` が付いていないことを特定
  3. 修正ブランチ `fix/keycloak-email-readonly` を作成
  4. 3ファイル（`LoginForm.tsx`, `EmailField.tsx`, `TextField.tsx`）を修正（readOnly属性追加・スタイル対応）

- **所要時間**: Slack投稿→修正完了まで約10分

### Keycloak固有の知見
- Keycloakからkeycloakifyでカスタムテーマを作る場合、ステップ間で状態を持つフォームは `readOnly` 属性の管理が重要
- パスワードマネージャーはページ上のinputフィールドを自動検出して上書きするため、email確定後のパスワード入力画面では email inputは必ず `readOnly` にする

## パターン4: クロスサービス・コードQ&Aの即時対応

### ユースケース
「このコードどういう意味？」「この実装の設計意図は？」をSlackで質問 → OpenClawがコードを読んで解説

### 効果
- エンジニアが他メンバーにコードを聞く待ち時間をゼロ化
- 複数ファイルをまたぐ実装も一度に把握
- 質問チャンネル（例: `#openclaw_request`）を設けると自然に活用が広がる

## 導入時のTips

### バグ報告チャンネルにOpenClawを招待する
`#report_bug` などのチャンネルにOpenClawを参加させ、メンションで呼び出す運用が効果的。
報告者がスクショ＋テキストを投稿するだけで、調査・修正・PR作成まで自動化できる。

### 調査系タスクの依頼フォーマット
```
@OpenClaw 調査依頼
[現象] mainのE2Eが落ちてる（{リポジトリURL}/actions/runs/{run_id}）
[期待] 原因コミットと修正方針を教えて
```

### 注意点
- `readOnly` / `disabled` の違いに注意: `disabled` はフォームの送信値に含まれないため、バリデーションやAPI送信で意図しない挙動になる場合がある。通常は `readOnly` を選択する。
- CIログが長い場合、OpenClawは重要な部分を抽出して提示する（全ログを読む必要なし）
