# OpenClaw コードレビュー対応の運用知見

_追加日: 2026-03-27_

## 知見1: 既存コードを修正する前に「なぜその書き方か」を先に理解する

### 背景
OpenClawが `taskhub-neo-frontend` のPR修正中に発生したインシデント。

### 問題
- 既存コードで `fileId = file.dataUrl`（`secure-vault://` 形式）と書かれていた
- 別の箇所の `url = file.previewUrl ?? file.dataUrl` と書き方が違うため「バグだ」と判断して修正してしまった
- 実際は `fileId`（送信用）と `url`（表示用）は役割が異なり、意図的に別の値が入っていた

### 学び
既存コードが「違う書き方」をしている場合の対処順序：
1. **まず「なぜ違うのか」を理解する** — コードコメント、テスト、型定義から読み取る
2. **読んでもわからなければ確認する** — 憶測で修正しない
3. **修正する場合はテストも必ず確認** — テストが既存仕様を証明していることが多い

### ルール化
```
既存コードの書き方が他と違う → バグではなく「意図的な差異」の可能性
→ コードを読んで意図を確認 → 確認できたら修正 → テストも合わせて更新
```

## 知見2: 知見蓄積を「仕組み」で担保する（手動に頼らない）

### 背景
OpenClawが運用中に得た知見を、手動でリポジトリに反映する運用にしていた。
→ 「気づいた時にやる」方式はサボりやすく、実際にサボっていた。

### 解決策
日報cronに知見の自動抽出・PR作成を組み込む：

```
毎朝8時 (日報cron実行)
  ↓
前日のセッションログ (*.jsonl) を読む
  ↓
OpenClaw運用知見を自動抽出
  ↓
openclaw-implementation-support にブランチ作成 & ファイル追記
  ↓
PR作成 → #147_times_openclaw に要約を投稿
```

### ポイント
- 「手動でやる」→「仕組みでやる」へ切り替えることで確実に蓄積される
- 知見があるかどうかの判断もAI（cron実行のOpenClaw）が行うため、人間の判断コストゼロ
- 1日1PR以内に収まるのでPR数が爆増しない

## 知見3: ファイルアップロード実装でのURL設計

### コンテキスト
AIチャット系のSaaSでファイルアップロード機能を実装する際のURL管理。

### パターン
```
dataUrl  = secure-vault://xxxx   → バックエンドへの送信用（永続ID）
previewUrl = https://s3.presigned → フロントエンド表示用（有効期限あり）
```

### 落とし穴
- `previewUrl` は有効期限（presigned URL）があるので保存して使い回せない
- `dataUrl`（`secure-vault://` 形式）は表示には使えない
- コンポーネントに渡すときは `url`（表示）と `fileId`（送信）を明示的に分けること
