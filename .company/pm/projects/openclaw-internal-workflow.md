# プロジェクト: OpenClaw社内活用ワークフロー

## 概要

OpenClaw + Slack + Mac mini（常時稼働）で、開発プロセスの各フェーズを効率化する。
Slackを窓口にして、自然言語で依頼 → OpenClawが処理 → 結果をSlackに返す。
Notionのステータス変更をトリガーに後続の自動化も行う。

## ゴール

社内の開発プロセス（要求管理〜運用保守）をOpenClawで効率化し、工数を削減する。

---

## フローA: 機能開発サイクル（要求 → リリース）

### 全体像

```
① 要求登録（Slack → OpenClaw → Notion）                    全自動
   Slack: 「○○の機能追加を登録して」
   → 類似要求チェック → 整形 → Notion要求DBに登録
   → Lv.2 ざっくり要件定義・工数見積を自動生成
────────────────────────────────────────────────────────────
② 着手決定（Notion）                                        PdM
   PdMがNotionで優先度判断 → ステータスを「着手決定」に変更
────────────────────────────────────────────────────────────
③ 開発準備（Notion webhook → OpenClaw → Notion + Slack）    全自動
   → 既存コード参照 → Lv.3 詳細要件定義に更新
   → 開発DBに親チケット生成（要求DBとリレーション紐づけ）
   → 子チケットを分割生成
   → QAリストを生成
   → テックリードにSlackメンション
     「チケット準備できました、アサインお願いします」
────────────────────────────────────────────────────────────
④ アサイン（Notion）                                        テックリード
   テックリードがチケットを各開発者にアサイン
────────────────────────────────────────────────────────────
⑤ 開発（Slack → OpenClaw → GitHub）                        開発者（+OpenClaw任意）
   各開発者の判断でOpenClawに依頼してもOK
   依頼方法: Slackから @OpenClaw
   品質責任: 開発者本人
────────────────────────────────────────────────────────────
⑥ コードレビュー（GitHub → OpenClaw → Slack → GitHub）      OpenClaw + 人間
   GitHub PR作成 → OpenClawが一次レビュー（自動）
   → 人間が最終レビュー・approve
────────────────────────────────────────────────────────────
⑦ QA実行（Slack → OpenClaw → Slack）                       OpenClaw + 担当者
   Slackから依頼: Notion要件定義URL + preview/ベータ環境URL
   → OpenClawがブラウザ操作でQA実行
   → 結果をスクショ/動画付きでSlackに報告
   品質責任: アサインされた担当者（親タスクはテックリード）
   NG時 → ⑤に戻る
────────────────────────────────────────────────────────────
⑧ PdMレビュー（Slack）                                     PdM
   Slackで以下を送付してレビュー依頼:
   - preview/ベータ環境URL
   - Notion要件定義URL
   PdMが確認・承認
   NG時 → ⑤に戻る（要件の認識ズレなら③から再検討）
────────────────────────────────────────────────────────────
⑨ リリース（GitHub）                                        人間
   PdM承認後、人間の手でマージ → リリース
────────────────────────────────────────────────────────────
⑩ リリース後処理（OpenClaw → Notion + Slack）               全自動
   → ビジネス仕様書を更新（※導入している場合のみ）
   → リリースお知らせの下書きをNotionに作成
   → bizにSlackでレビュー依頼
   → bizが確認・配信
```

### 要件定義レベルの定義

| レベル | 内容 | 誰が/いつ |
|-------|------|----------|
| Lv.1 要求登録 | 「○○したい」を整形して記録 | OpenClawが自動（①） |
| Lv.2 ざっくり要件定義 | 機能要件・影響範囲・ざっくり工数 | OpenClawが自動（①と同時） |
| Lv.3 詳細要件定義 | 非機能要件・エッジケース・詳細工数・変更範囲 | OpenClawが自動（③、着手決定後） |

### 役割分担サマリー

| フェーズ | OpenClaw | 人間 | 判断背景 |
|---------|:--------:|:----:|---------|
| ① 要求登録〜Lv.2 | ◎ | 確認のみ | 定型的な整形・分析作業 |
| ② 着手決定 | × | ◎ PdM | ビジネス優先度は人間の判断 |
| ③ Lv.3〜チケット・QA生成 | ◎ | 確認のみ | 着手確定後の展開作業 |
| ④ アサイン | × | ◎ TL | 稼働・スキル・成長機会の判断 |
| ⑤ 開発 | △ 任意 | ◎ | 各開発者の判断で使い分け |
| ⑥ レビュー | ◎ 一次 | ◎ 最終 | 機械的チェック→人間の判断 |
| ⑦ QA | ◎ 実行 | ◎ 責任 | 自動実行するが品質責任は人間 |
| ⑧ PdMレビュー | × | ◎ PdM | ビジネス要件の最終確認 |
| ⑨ リリース | × | ◎ | マージ・リリースは人間の手で |
| ⑩ リリース後 | ◎ | biz確認 | 仕様書更新・お知らせ下書きは自動化 |

### 開発の自動化について（段階的に導入）

全チケットをOpenClawに開発させるのは現時点ではやりすぎ。品質担保が難しく、レビュー負荷が逆に増えるリスクがある。

**段階的アプローチ:**
1. まずチケット・QA生成までの自動化を完成させる
2. 運用が安定したら「このチケットはOpenClawに開発させる」をテックリードが判断できるようにする
3. 最初は簡単なチケット（依存関係更新、定型CRUD、テストコード生成など）から試す

### トリガーの仕組み

Notionデータベースオートメーションの **Send webhook** 機能を使用。

```
Notionオートメーション設定:
  トリガー: ステータスが「着手決定」に変更
  アクション: Send webhook → OpenClawのエンドポイントにPOST
```

- Notion有料プランが必要（社内利用なので問題なし）
- OpenClaw側はMac miniで常時稼働しているのでWebhook受信可能

## フローB: バグ修正

```
Slackバグ修正チャンネルで @OpenClaw に依頼
  → OpenClawが情報収集（ログ、再現手順、関連コード）
  → 修正PR作成
  → Slackで報告 + レビュー依頼
  → 人間がレビュー・マージ
```

- 依頼から修正PRまでOpenClawに一任
- レビュー・マージは人間が実施

---

## フローC: 仕様確認（biz・顧客対応）

```
Slackで @OpenClaw 「○○の仕様ってどうなってる？」
  → 即回答
```

- ビジネス仕様書を導入した場合はそちらを参照させる（コスト・精度が向上）
- ビジネス仕様書の導入判断後に参照先を再検討する

---

## フローD: エラー検知 → 自動修正

```
Sentry / OpenSearch からアラート通知
  → OpenClawが自動で原因調査
  → 修正PR作成
  → Slackでメンション（※メンション先は要検討）
  → 人間がレビュー・マージ
```

- メンション先の決め方は今後検討（稼働中の人に自動メンションなど）

---

## フローE: 死活監視

```
Cron（定期実行）
  → OpenClawがブラウザでWebアプリにアクセス
  → 正常: ログ記録のみ
  → 異常: スクショ付きでSlackアラート
```

- 監視間隔・監視対象URLは設定で管理
- 異常検知時のエスカレーション先は要検討

---

## フローF: 依存関係更新

```
定期スキャン（Cron）
  → パッケージの更新を検知
  → 互換性チェック（テスト実行）
  → 更新PR作成
  → Slackで報告 + レビュー依頼
  → 人間がレビュー・マージ
```

- Dependabotの代替として運用
- テストが通らない場合は修正案も含めてPRに記載

## 検討事項

### ビジネス仕様書の導入（2026-03-22 検討中 → 導入方向で提案予定）

**方針: 導入する方向で検討。フェーズ2でマネージャー陣に提案する。**

**導入すべき理由:**
フローA①のLv.2要件定義で「既存仕様との照合」「デグレリスク洗い出し」をやるには、既存仕様を知る必要がある。コードを読んでも「どう実装したか」はわかるが「何をどういう意図で作ったか」はわからない。まともな要件定義を自動生成するにはビジネス仕様書が前提になる。

**各業務への影響:**

| 業務 | 仕様書あり | 仕様書なし | 備考 |
|------|:--------:|:--------:|------|
| ① Lv.2 要件定義 | ◎ | △ コード読みでは意図がわからない | 仕様書がないとまともなLv.2が書けない |
| ② QAリスト生成 | ◎ | ○ 仕様ベース+コード | 仕様書があると精度向上 |
| フローC 仕様確認 | ◎ | × | 仕様書がないと成り立たない |
| ③ Lv.3 チケット生成 | △ | ◎ コード直接 | 変更範囲の詳細はコードが正 |
| ⑩ リリース後更新 | ◎ 自動更新 | - | 仕様書があれば自動で最新化 |

**仕様書の位置づけ:**
- コードの実装詳細ではなく「何ができるか・なぜそうしたか」のビジネス視点要約
- 位置づけは **意思決定の記録**（コードの派生物ではない）→ 結合度を低く保つ
- 更新サイクルはリリース単位（フロー⑩で自動更新）
- 初期生成はOpenClawに既存コードから作らせて、人間がレビュー・修正

## 技術メモ

- Mac mini常時稼働: `openclaw --install-daemon`、Cloudflare Tunnel（無料）
- ブラウザ操作: Playwright（CDP）ベース。動画録画・スクショ対応
- 複数リポジトリ: 1エージェント + GitHub MCP方式が最もシンプル
- Notionトリガー: Database Automations + Send webhook（有料プラン必要）
- セットアップ工数目安: 合計5.5〜9.5時間（1〜2日）
