# OpenClaw × Slack 社内活用ワークフロー計画

## 概要

OpenClaw + Slack + Mac mini（常時稼働）で、開発プロセスの各フェーズを効率化する。
Slackを窓口にして、自然言語で依頼 → OpenClawが処理 → 結果をSlackに返す。

---

## 効率化する業務

### 1. 要求管理

| 現状の課題 | OpenClawで解決 |
|-----------|---------------|
| 要求が重複する | 登録時に類似要求を自動チェック、統合を提案 |
| 内容がわかりにくい | 自然言語の入力を整形して登録 |

```
Slack: 「○○の機能追加を登録して」
  → 類似要求チェック → 整形 → Notionに登録
```

### 2. 要件定義の自動生成

要求リストから機能要件・非機能要件を自動生成。既存仕様と照合し、デグレリスクや競合箇所を洗い出す。

### 3. QAリスト生成・デグレ確認

仕様ベースでQA項目を自動生成。ブラウザ操作で実際に確認し、結果をスクショ付きでSlackに報告（NGの場合は動画も可）。

### 4. 仕様確認（biz・顧客対応）

Slackで「○○の仕様ってどうなってる？」→ 即回答。biz・顧客からの問い合わせに人間が調べる手間を削減。

### 5. 開発チケット生成

要件定義からチケットを自動生成。変更範囲・影響範囲は既存コードを参照して詳細を記載。

### 6. コードレビュー一次対応

PRの自動レビュー → 指摘事項をSlackに報告 → 人間が最終判断。

### 7. 運用・保守の自動化

- **アラート対応**: Sentry等の検知 → 原因調査 → 修正PR or 報告
- **依存関係更新**: パッケージ更新検知 → 互換性チェック → 更新PR作成
- **死活監視**: 定期的にブラウザアクセス → 異常検知 → Slack報告

### 8. その他

- 工数見積（要件・チケットから自動算出）
- 技術リサーチ（Slackで依頼 → 調査 → ナレッジに蓄積）
- リリース配信（承認フロー付き。自動実行は危険なので人間承認を必須とする）

## 検討事項

### 仕様書を用意すべきか（2026-03-22 結論済み）

**結論: いったんなしで着手。問題が出たら導入する。**

仕様書があると助かる場面（2, 3, 4）と、コード直接参照のほうがよい場面（5）がある。

| 業務 | 仕様書 | コード直接 | 理由 |
|------|:-----:|:---------:|------|
| 要件定義生成 | ◎ | △ | コード全量読みはコスト過大 |
| QAリスト生成 | ○ | ○ | 仕様ベース + コードのエッジケース |
| 仕様確認 | ◎ | × | コード参照はオーバーキル |
| チケット生成 | △ | ◎ | 変更範囲の詳細が必要 |

コストや精度に問題が出たタイミングで、既存コードから **「ビジネス仕様書」** を作成する。

- コードの実装詳細ではなく「何ができるか・なぜそうしたか」のビジネス視点要約
- 位置づけは **意思決定の記録**（コードの派生物ではない）→ 結合度を低く保つ
- 更新サイクルはリリース単位でOK

## 着手順序

1. Mac miniにOpenClawセットアップ → Slack連携
2. **要求リスト管理** から試す（最も相性がよく、効果が見えやすい）
3. 運用しながらナレッジを貯める → 記事ネタにもなる
4. 他のワークフローを順次追加

## 技術メモ

- Mac mini常時稼働: `openclaw --install-daemon`、Cloudflare Tunnel（無料）
- ブラウザ操作: Playwright（CDP）ベース。動画録画・スクショ対応
- 複数リポジトリ: 1エージェント + GitHub MCP方式が最もシンプル
- セットアップ工数目安: 合計5.5〜9.5時間（1〜2日）
