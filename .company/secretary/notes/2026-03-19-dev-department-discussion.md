# 開発部署の設置と事例活用に関する検討メモ

**日付**: 2026-03-19
**起案**: 秘書メモ（別ターミナルでの深掘り用）

---

## 1. 発端

### リポジトリ内に開発部署を作る案の見解

このリポジトリ（openclaw_implementation_support）は**マーケティング + 事業管理**を目的として構築されている。ここに開発部署を追加して実際の開発作業まで行うのは、以下の理由から厳しいという作成者の見解がある：

- リポジトリの設計思想がマーケ・管理に特化している
- 開発コードとマーケ資産の混在は運用上の問題を生む
- Claude Codeのコンテキスト管理が煩雑になる

### しゅーへいさんの関心

「**開発の手前〜開発までを効率化したい**」というクライアントへの事例として、このリポジトリの運用パターンが活用できないか？という視点。

## 2. このリポジトリが事例として使える部分

### 仮想組織（.company/）の運用パターン

- **秘書 → CEO → 部署**の振り分けフロー
- TODO管理の仕組み（secretary/todo/）
- ナレッジの蓄積と再利用（各部署のknowledge/）
- 組織構造をディレクトリで表現し、Claude Codeが自律的に動く仕組み

### SEO記事ワークフロー

- `/article` スキルによる**8フェーズ**の記事作成プロセス
- revisions/ によるリライト管理
- 構成案 → 執筆 → レビュー → 公開準備の体系化

### Claude Codeのカスタムスキル活用

- 業務の仕組み化（繰り返し作業のスキル化）
- 属人性の排除と品質の安定化
- 非エンジニアでも使える業務自動化の実例

## 3. 開発を含む場合の課題

### リポジトリ分離が必要

マーケ/管理と開発コードは別リポジトリにすべき。1つのリポジトリに混在させると以下の問題が発生する。

### コンテキスト汚染

- 記事ナレッジとソースコードが混在すると、Claude Codeの効率が低下する
- CLAUDE.mdの指示が肥大化し、意図しない動作を引き起こすリスク
- 記事執筆時にコードの知識が邪魔になり、その逆も同様

### git運用の違い

| 項目 | マーケ/管理リポ | 開発リポ |
|------|----------------|----------|
| ブランチ戦略 | main直push | feature → PR → merge |
| コミット粒度 | 記事単位 | 機能・修正単位 |
| レビュー | 記事レビュー | コードレビュー |
| CI/CD | 不要 | 必須 |

### 権限・公開範囲の違い

- マーケリポ: 公開可能な情報が中心（SEO記事など）
- 開発リポ: 機密情報を含む可能性（APIキー、内部ロジック等）
- 公開リポ vs プライベートリポの使い分けが必要

## 4. クライアント提案の方向性案

### パターン紹介: 「仮想組織 + カスタムスキル」

- このリポジトリの`.company/`構造をテンプレートとして紹介
- Claude Codeのカスタムスキルによる業務仕組み化の事例を提示
- 「開発の手前」（要件整理、仕様策定、タスク管理）はこのパターンで対応可能

### 開発を含む場合: 2リポ構成の提案

```
クライアント組織/
├── management-repo/     ← マーケ・管理（このリポジトリのパターン）
│   ├── .company/
│   ├── articles/
│   └── ...
│
└── development-repo/    ← 開発コード
    ├── src/
    ├── tests/
    └── CLAUDE.md        ← 開発用の指示
```

- 管理リポで要件整理 → 開発リポで実装、という連携フロー
- それぞれのCLAUDE.mdを最適化し、コンテキスト汚染を防ぐ

## 5. 実行エンジンの選択肢: Claude Code vs OpenClaw

### 3パターンの比較

| パターン | 構成 | 現実性 |
|----------|------|--------|
| A. OpenClaw → Claude Code → リポジトリ | OpenClawがClaude Codeを呼び出す | ✕ 繋ぎ込みが複雑 |
| B. Slack → Claude Code → リポジトリ | webhook経由でClaude CLIを叩く | △ 自前開発が必要 |
| C. OpenClaw → リポジトリ直接操作 | OpenClawがAgentSkillでファイル操作 | ◎ 最も自然 |

### 結論: パターンCが最も筋がいい

- OpenClawはSlack/Teams**常駐が本業**（Claude Codeはローカル実行ツール）
- OpenClawもAgentSkill経由で**ローカルファイルの読み書きが可能**
- AとBは「繋ぎ込み」の開発が必要だが、Cは**単体で完結**する

```
ユーザー → Slack/Teams → OpenClaw → .company/（このリポジトリ）
```

### この構造が実行エンジンに依存しない理由

- **ディレクトリ構造** = 業務フローの設計図（秘書→CEO→部署の振り分け）
- **CLAUDE.md** = 各ロールの振る舞い定義 → OpenClawのAgentSkillの`systemPrompt`に転用可能
- **テンプレート** = ファイル命名規則やTODO形式 → そのまま使える
- **スキル（/article等）** = ワークフロー定義 → AgentSkillのタスク定義に変換可能

実行エンジンがClaude CodeからOpenClawに変わっても、「何をどう処理するか」のロジックは同じ。今Claude Codeで試行錯誤しながら構造を固めておくと、OpenClaw移行時の設計がすでにできている状態になる。

### クライアント提案への示唆

- 「まずClaude Codeで業務フロー設計を固める → OpenClawに載せ替えてチャット常駐化」という段階的アプローチが提案できる
- `.company/`構造とスキル設計は**プラットフォーム非依存の業務設計資産**として位置づけられる
